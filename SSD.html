Secure Software Development

This page gathers weekly artefacts, short reflections, and the three summative assessments for the Secure Software Development module. The focus is applied: secure design, programming fundamentals, testing discipline, cryptography, APIs, and system architecture — and how these come together to ship software that is safe, reliable, and maintainable.

Main goals for the module

  1. Build a critical understanding of abstraction as a programming concept.
  2. Grasp the core principles underpinning secure development methodologies.
  3. Apply analysis, design, construction, and testing techniques across the software lifecycle.
  4. Understand software architecture and SDLC models, from traditional frameworks like TOGAF to Agile.
  5. Put techniques into practice in a team setting — managing conflict, making trade-offs, and evaluating outcomes.
  6. Reflect on personal development and identify areas for continued growth.

01 UNIT: Intro to Secure Software Development

This unit introduced the foundational principles of building secure software, contrasting the limitations of traditional approaches like Waterfall with the strengths of Agile. Agile's iterative rhythm and collaborative ethos make it better suited to dynamic, security-sensitive environments.

Activity: I used Umbrello, an open-source UML tool, to produce a flowchart tracing the steps that lead to a common OWASP vulnerability — a practical way to see how UML clarifies complex system interactions.

Read more

The unit also reinforced that secure software is not just about code: it requires a risk-aware culture where developers and stakeholders actively look for and address threats. Establishing that mindset early is essential to sustaining secure practices throughout the lifecycle.

Team-wise, we formed our project group: set up a WhatsApp channel for daily coordination, signed a Team Contract, and created a shared Google Drive workspace. These foundations gave us clear communication channels, defined responsibilities, and a single source of truth for collaborative work.

02 UNIT: UML Modelling for Secure System Planning

This unit developed my UML modelling skills for secure system design within the Agile SDLC. UML diagrams and flowcharts act as a shared language between stakeholders — grounding the design phase and carrying through to implementation.

Seminar: I produced a table mapping security recommendations to each Scrum stage (drawing on Sharma and Bawa's work) and wrote a blog post on insider cyber risk grounded in five terms from ISO/IEC 27000 Section 3.

Read more

A central theme was embedding security into every SDLC phase rather than bolting it on. I practised use case, sequence, activity, and class diagrams to visualise how security considerations ripple through both structure and workflow. Grounding these models in the ISO/IEC 27000 series kept the terminology consistent with industry standards.

In our first online team meeting, we weighed several application domains and selected Online Retailer as our project focus. We organised tasks in Trello and set weekly Sunday stand-ups, giving the team a predictable cadence for progress and blockers.

03 UNIT: Programming Languages — History, Concepts & Design

This unit traced the evolution of programming languages from early machine code through COBOL to modern Python, recognising the influence of figures like Alan Turing. Using Python, I practised inheritance, polymorphism, and abstraction — the pillars of OOP.

Activity: I compared buffer overflow handling in C and Python, and implemented a producer-consumer model with thread synchronisation using queues to prevent data loss and duplication.

Read more

A strong focus of the unit was secure coding practice and common vulnerabilities. I used Pylint to check code quality and style, and reflected on how Python's memory safety gives it a structural advantage over C for many secure development contexts.

Team discussions compared Python and C for OS-level work and explored how to harden the producer-consumer model. Our second team meeting also progressed the secure software design document, helping align everyone's view of the architecture.

04 UNIT: Exploring Programming Language Concepts

This unit zoomed in on two powerful yet risky concepts: regular expressions (RegEx) and recursion. Both enable elegant solutions but carry real security implications if misused.

Seminar: I examined Regular Expression Denial of Service (ReDoS) attacks caused by "Evil Regex" patterns, and identified mitigations — simplifying patterns, enforcing timeouts, and pre-testing expressions for safety.

Read more

Recursion — where a function calls itself to solve problems like traversal, searching, or sorting — is similarly double-edged. Without a clear base case or tail recursion optimisation, it risks stack overflows and degraded performance, so I looked at the safeguards that make recursive solutions production-ready.

The broader lesson was that inefficient or overly complex implementations of either concept can produce denial-of-service risks or performance failures. Our third team meeting continued work on the design document, tightening the link between theory and our practical API design.

05 UNIT: Introduction to Testing

This unit introduced software testing as a discipline covering both general quality and security. I explored industry-standard processes from OWASP and ISO/IEC/IEEE, and practised different testing approaches — functional, usability, performance, penetration, and vulnerability testing.

Activity: I worked with equivalence partitioning (equivalence.py) to split input data into classes for systematic test coverage, and used Python tools including Bandit and other linters for automated checks.

Read more

A further focus was cyclomatic complexity and its role in modern secure development. I evaluated its usefulness for judging code design and secure maintainability, which sharpened my ability to critically assess when a metric genuinely informs decisions versus when it becomes noise.

The unit equipped me to manage security risk systematically throughout a project and to select the right methodology, tool, or technique for the challenge at hand — moving testing from a ceremony to a decision-making tool.

06 UNIT: Using Linters to Support Python Testing

This unit focused on linters as a practical control for producing high-quality, secure Python code. I worked with Pylint, Flake8, and McCabe — each offering a different lens on error detection, style enforcement, and complexity measurement.

Team milestone: We finalised and peer-reviewed the Development Team Project Design Document — a secure, CRUD-functional RESTful API proposed for internet forensics — which forms Summative Assessment 1.

Read more

Running each tool against different scripts gave me hands-on experience correcting errors and reducing complexity. Pylint surfaced best-practice violations, Flake8 streamlined style and error checks, and McCabe highlighted where complex control flow made code harder to maintain — and therefore harder to keep secure.

Team coordination was continuous this week: an online meeting on 1 December plus ongoing WhatsApp communication kept us aligned as we closed out the document. The combination of automated linting and peer review raised the overall quality bar noticeably.

07 UNIT: Introduction to Operating Systems

This unit explored the fundamentals of operating systems and their relationships with programming languages and software security. I covered OS types and functions, the differences between processes, threads, and schedulers, and the distinction between static and dynamic libraries.

Activity: I built a simple Python CLI shell supporting directory listing, basic arithmetic, and help commands — then analysed it for vulnerabilities and proposed input-validation improvements.

Read more

I also built a RESTful API with Flask for a distributed environment, supporting user record creation, deletion, and modification. Running and testing it gave real insight into how libraries like Flask enable web-based interactions while making secure API design a non-negotiable concern.

The unit also introduced ontologies and their relevance to software systems, prompting me to think about how a structured vocabulary could strengthen the secure CRUD-functional RESTful API our team was designing for the summative assessment.

08 UNIT: Cryptography and Its Use in Operating Systems

This unit focused on cryptography and its role in securing modern systems — protecting data, preserving privacy, and maintaining the integrity of communications. I examined common cryptographic libraries and their real-world OS applications.

Seminar: I wrote a Python program that encrypts text files using a chosen cryptographic algorithm, selected on strength and implementability, and evaluated whether the approach aligns with GDPR's privacy-by-design expectations.

Read more

For the collaborative discussion, I analysed the TrueCrypt case study — its vulnerabilities and user exposure — and considered how ontology can help identify key weaknesses and shape mitigation design.

The combined practical and case-study work gave me a clearer view of how cryptographic choices must be justified against both technical standards and regulatory requirements — not treated as a library import.

10 UNIT: From Distributed Computing to Microarchitectures

This unit surveyed modern system architectures — from monolithic deployments to distributed systems, microservices, virtualisation, and containers — and looked at how each shift expands the attack surface alongside its scalability benefits.

Activity: I reviewed Schmitz et al. (2016) on faceted data as a mechanism to prevent data leakage, and sketched a basic Python outline showing how controlled views could enforce least-privilege access programmatically.

Read more

Secure communication across decentralised components stood out as a core concern, and I explored techniques like encryption and secure key distribution alongside the specific vulnerabilities that show up in virtualised environments.

Faceted data proved a good lens on the trade-offs: strong for minimising exposure, but needing careful implementation to avoid performance bottlenecks or unnecessary complexity. Overall the unit tied architecture choice directly to security posture.

11 UNIT: Future Trends in Secure Software Development

This unit looked at emerging directions in secure development — including Fog Computing, the Internet of Things, and Cyber Physical Systems — and the specific risks they introduce: data privacy concerns, authentication weaknesses, and network-level vulnerabilities. I also reviewed the Tanenbaum-Torvalds debate as a lens on architectural futures.

Assessment 2: For the Development Individual Project I built a secure B2B e-commerce CLI application — a CRUD-functional system with centralised logging, input validation, exception handling, encryption, and multi-threading for concurrent operations.

Read more

Engineering quality was a first-class concern: automated linting, unit and integration tests, dependency vulnerability scanning, and a shared conda virtual environment to keep the team's tooling consistent. The application was packaged as a pip-installable Python module to simplify deployment.

Combining theory with hands-on delivery made the emerging trends feel tangible — not abstract predictions, but concrete design decisions that change how secure software should be planned, built, and shipped.

12 UNIT: The Tanenbaum-Torvalds Debate Revisited

This unit revisited the historic Tanenbaum-Torvalds debate, weighing monolithic kernels against microkernels and microservices in the context of today's multi-core, virtualised, cloud-native systems — where cyber-attack surface and modularity demands have both grown dramatically.

Final Reflection: Drawing on Biggs et al. (2018) and Bucchiarone et al. (2018), I took a position that microservices and microkernels are the more future-proof choice, trading some raw performance for flexibility, modularity, and fault isolation in distributed environments.

Read more

In the forum discussions I posted justifications for our team's stance and responded to counter-arguments from opposing teams. I also contributed to the team's evaluation of modern secure programming advances — including faceted data approaches and their role in system design.

The unit was a strong close to the module: it pulled together architecture, security, and practical delivery into a single argument about where the field is heading, and forced clear reasoning about the trade-offs behind each option.
